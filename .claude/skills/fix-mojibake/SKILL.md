# fix-mojibake — 日本語文字化け修正

## 目的

「文字化けしたテキスト」を **目視推測で書き直さず**、エンコーディング変換で **可逆に復元**する。
原文を捨てて当てずっぽうで直すと、別の意味に化けたまま PR や DB に入り込むので、**変換ルートを特定 → iconv/Python で戻す → 差分を見せて確認** の順で進める。

## トリガ（このスキルを呼ぶとき）

- ユーザーのメッセージ・貼り付け・選択範囲に下記のいずれかが混じっている
  - ひらがな・漢字の中に **半角カタカナ片** が混じる：`ｱ ｲ ｳ ﾊ ﾞ ﾟ ｦ ﾝ` など単独で出る
  - **「縺」「繧」「繝」「ﾃ」「ﾂ」** で始まる連結文字列（例: `縺ゅ→縺ｮ縺ｾ縺ｾ`）
  - **「ï½」「ã」「â」** で始まる連結（例: `ãã®ã¾ã¾`、`ï½žï½`）
  - **「・」「･」「ｿ」「?」** が文中にランダムに散らばる
  - 単語の中に **本来あり得ない漢字** が紛れる（例: 「作成」が「割く成」「割くセ」「セ氏」、「修正」が「修．」など）
- ユーザーが「文字化け」「mojibake」「直して」と明示
- 上記の混入を含む文字列を **コード/ファイルに書き込もうとしている**（書き込み前に止める）

## 必ず守ること

1. **目視で書き換えない**。`割くセ氏て → 作成して` のような置換を**勘で**やらない。
   - 1文字でも誤ると別の意味に化ける。Diff レビューで気づけない。
2. **元の文字列を必ず保存**してから変換する。失敗時に戻せるようにする。
3. **変換ルートを2〜3個試して、最も自然な日本語になったものを採用**する。複数候補が成立する場合はユーザーに確認。
4. ユーザーの選択範囲・貼り付けに含まれるだけのときは、**ファイル書き込みは行わず**、復元結果をテキストで返すだけにする。

## 進め方

### Step 1 — パターン判定（化け方の分類）

化けたサンプルを 1〜2 行コピーして、下の見分け表で **どのルートで化けたか** を当てる:

| サンプルの見た目                               | 推定ルート                                                          | 復元コマンド                                                                   |
| ---------------------------------------------- | ------------------------------------------------------------------- | ------------------------------------------------------------------------------ |
| `縺ゅ→縺ｮ縺ｾ縺ｾ` のように `縺/繧/繝` が頻発    | **UTF-8 を Shift_JIS として解釈してしまった**                       | `iconv -f UTF-8 -t SHIFT_JIS \| iconv -f UTF-8 -t UTF-8` 相当（Python が安全） |
| `ãã®ã¾ã¾` のように `ã/â/ï` が頻発              | **UTF-8 を Latin-1 (ISO-8859-1) として解釈してしまった**            | `bytes.decode('utf-8').encode('latin-1').decode('utf-8')` 相当（後述）         |
| `ï½žï½ï½` のように `ï½` が連続                 | **UTF-8 → CP1252 → UTF-8 の二重エンコード**                         | `latin-1` ルートを 2 回かけることがある                                        |
| `ｱｲｳｴｵ` のように半角カナだけが残る             | **全角→半角化された / Shift_JIS の半角カナ域だけ拾った**            | 半角→全角変換（`jaconv` 等。文脈次第で不可逆）                                 |
| `割くセ氏て` のように **漢字が別漢字に化ける** | **OCR / 音声入力 / 予測変換の誤り**（エンコーディング起因ではない） | **このスキルでは直さない**。ユーザーに原文を聞き直す                           |

> 重要: 最後のパターン（漢字レベルの変換ミス）は **エンコーディング由来ではない**。
> iconv では復元できない。勝手に書き直さず、ユーザーに **原文を確認**する。

### Step 2 — 復元コマンドの実行

判定したルートに応じて、**Bash 上の Python ワンライナー**で復元する。
`iconv` 単発だと壊れる/止まるケースが多いので、Python の `bytes.decode/encode` を基本にする。

化けサンプルを `/tmp/mojibake.txt` に置いてから:

```bash
# パターンA: UTF-8 を Shift_JIS と誤読
python3 -c "import sys; s=open('/tmp/mojibake.txt',encoding='utf-8').read(); print(s.encode('cp932','ignore').decode('utf-8','replace'))"

# パターンB: UTF-8 を Latin-1 と誤読
python3 -c "s=open('/tmp/mojibake.txt',encoding='utf-8').read(); print(s.encode('latin-1','ignore').decode('utf-8','replace'))"

# パターンC: UTF-8 を CP1252 と誤読（Latin-1 の上位互換、Windows 系で頻出）
python3 -c "s=open('/tmp/mojibake.txt',encoding='utf-8').read(); print(s.encode('cp1252','ignore').decode('utf-8','replace'))"

# パターンD: 二重化け（B または C を2回かける）
python3 -c "s=open('/tmp/mojibake.txt',encoding='utf-8').read(); print(s.encode('latin-1','ignore').decode('utf-8','replace').encode('latin-1','ignore').decode('utf-8','replace'))"
```

`errors='replace'` で残る `�` の数が少ない／日本語として読めるものを採用する。

### Step 3 — 復元結果の確認

- 復元後テキストを **そのまま** ユーザーに見せる（要約しない）。
- 「化けた箇所」だけでなく、**化けていなかったはずの周辺文字**まで Python 変換で壊れていないか必ず確認する（特に英数字・絵文字・引用符）。
- 2 ルート以上で読める日本語になった場合は、両方並べて `AskUserQuestion` で選んでもらう。

### Step 4 — 書き込み（必要な場合のみ）

- ファイル内の文字化けを直すよう依頼された場合のみ書き戻す。
- 書き込み前に: 元ファイルの該当箇所を Read で確認 → 復元結果と diff を提示 → Edit で置換。
- **コミット前の差分は必ずユーザー確認**。文字化け修正は見た目で気づきにくいので、自動コミットしない。

## やってはいけないこと

- 化け文字を **目で当てて書き直す**（`割くセ氏て` を勘で `作成して` に直す等）。
- `git commit -am "fix mojibake"` のような **無確認の一括コミット**。
- ターミナルやエディタの **表示が化けているだけ** のケースで、実ファイルを書き換える。
  → 先に `file -I <path>` / `nkf -g <path>` でファイル自身のエンコーディングを確認。
- バイナリ・画像ファイル等への適用。

## よくある原因の覚え書き

- macOS のターミナルで `LANG` が未設定 / `C` のとき、SSH 越しの貼り付けで Latin-1 ルートが発動しがち
- Slack / Notion / Google Docs からの貼り付けで CP1252 ルートが入ることがある
- Windows 環境で書かれたシェルスクリプトを mac で実行するときに Shift_JIS ルートが残ることがある
- GitHub の PR 本文や Issue タイトルが化けて見える場合、**ブラウザ側の表示問題**のことが多い。URL に `?_pjax=` などのキャッシュ回避を付けて再読込してから判断する

## このプロジェクトでの注意

- `apps/mobile`（Expo）のソース文字列は **UTF-8 LF 固定**。BOM 付与禁止
- `apps/backend`（Workers）でのレスポンス文字列は `Content-Type` に `charset=utf-8` を明示
- DB（Supabase / PostgreSQL）に流れる日本語は **必ず UTF-8 のまま** insert する。クライアント側の Shift_JIS 変換は経由させない
- コミットメッセージ本文の日本語は UTF-8。`/commit` スキルと併用するときは、本スキルで復元 → `/commit` でメッセージ整形、の順
